# Manifesto

```
1. "Entre antes se comience a trabajar, más rápido se encuentran los errores para poderlos corregir".
2. Guiar a las personas a norma general, se tienen 4 valores y 12 principios
3. El proposito es sentar bases guia.
```

# Mi primer product backlog

```
Es cómo un panel donde se registran tareas, donde ves de una manera visual, cómo gestionar tus progresos.

Un product Owner debe estar pendiente del product backlog
```

# Primer valor

```
1. Personas e interacciones >  procesos y herramientas
2. Respetar más a las personas y sus interacciones.
3. Ver a cada persona que es lo que necesita para realizar el trabajo. Escuche a los trabajadores para saber que le falta.
4. Logre soluciones, no embuta las cosas.
```

# Segundo valor

```
1. Producto funcionando > sobre documentación exhaustiva y/o extensiva
2. La unica medida real de progreso en agile es el producto funcional a traves de los MVP
3. Aún cliente el valor que le aporta un papel es cero.
4. Haga que la gente tenga el producto rápido.
5. El producto que se tiene con el producto es lo que realmente importa.
6. Centrarnos en traer cosas útiles para la empresa, trabajadores o clientes.
7. Ciclo ODDA <- Util para recibir feedback
```

# Tercer valor

```
1. Colaboración con el cliente > negociación de contratos
2. Predisposición a resolver situaciones, actitud win/win permanentemente
3. Ayude al cliente, sin necesidad de burocracia ineficiente.
4. Proponer al cliente que trabajameos conjuntamente.
5. Fomentar la capacidad de colaborar.
6. Darle la bienvebida a los cambios.
```

# Cuarto valor

```
1. Responder al cambio > seguir un plan
2. Centrarse en generar valor en entornos VUCA.
3. Si nos preparamos al cambio, nos estamo preparando a lo impredecible.
4. Las empresas estan trabajando con respecto a planes predictivos y no adaptativos.
5. Se tenga en cuenta que van a ocurrir situaciones que cambiaran las circunstancias.
6. Evitar poner topes, empiece y vea como se mueven las cosas.
7. Siempre habrá riesgo.
```

# Primer principio

```
1. La mayor prioridad es satisfacer al cliente mediante la entrega temprana y continuada de productos con valor con respecto a lo que necesita.
2. Lo más importante para el cliente que se le resuelva la necesidad rápido.
3. Se requieren ciclos iterativos/incrementales con feedbacks del Stakeholder, para sabr cómo trabajar.
4. La persona necesita un producto que entregue un valor.
```

# Segundo principio

```
1. Se acepta que lo requisitos cambien, incluso en etapas tardías del desarrollo.
2. Los procesos Agiles aprovechan el cambio para proporcionar ventaja competitiva al cliente.
3. Se hay una capacidad de crear planes hasta que se encuentre el apropiado.
4. La burocracia puede matar decisiones.
```

# Tercer principio

```
1. Se entregan productos funcionales con frecuencia, entre dos semana y un mes, con preferencia por periodos de tiempo lo más corto.
2. Se hace lo más corto posible de acuerdo al producto, debido a que se puede utilizar rápidamente en el mercado.
3. Poder recibir feedback rápidamente.
4. Procesos de colaboración permanente.
```

# Cuarto principio

```
1. Los responsables del negocio y los desarrolladores deben trabajar juntos de forma cotidiana durante todo el proyecto.
2. Se debe trabajar de forma horizontal y unida.
3. La jerarquia hace que los de arriba pierdan perspectiva.
4. Se debe trabjar en conjunto.
5. Se dialoga de manera abierta. Quite las barreras de la comunicación.
6. Que el PO no sea una visita, que no deje el trabajo y se abra. Ver si necesita ayuda.
7. Se acompaña a las personas para que se cumplan las metas
```

# Quinto principio

```
1. Los proyectos se desarrollan en torno a individuos motivados. Hay que darle el entorno y el apoyo que necesitan, y confiarles la ejecución del trabajo.
2. La voz de cada uno es escuchada y respetada.
3. Nos debemos de empoderar.
4. Hacer que las personas tengan ganas de desarrollar sus habilidades.
5. Las mejoras ideas salen de equipos autorganizados.
6. Entornos comodos para explotar potencial. Haga que las personas se muevan en lo que son buenas.
7. Una de las principales causas de abandono de un puesto es por falta de respeto, falta de motivación o no sentirse valorados.
8. Demuestre que la gente puede hacer cosas mejor. Que el trabajo no sea aburrido, si no que la gente pueda explotar su creatividad.
9. Que los de abajo se puedan abrir con los de arriba, y darles las oportunidades. 
```

# Sexto principio

```
1. el método más eficiente y efectivo de comunicar información al equipo de desarrollo y entre sus miembros es la conversación cara a cara.
2. Mantenga o simule conversaciones fluidas.
3. Evite formularios o emails, que entorpece procedimientos.
4. Vía de comunicación franca, directa y honesta. No significa ser grosero.
```

# Septimo principio

```
1. el producto funcionando es la principal medida de progreso.
2. El producto es lo único que resuelve el problema.
3. Es muy probable que los clientes no tengan un conocimiento profundo de lo que esta pidiendo.
4. No hable de cosas de la que el cliente no entienda.
```

# Octavo principio

```
1. Los procesos ágiles promueven el desarrollo sostenible. Los promotores, desarrolladores y usuarios debemos ser capacer de mantener un ritmo constante de forma indefinida. Trabajar no es el objetivo; alcanzar el producto es el objetivo.
2. Acelerar los tiempos crea estrés en el equipo y sobre carga all equipo.
3. Si ingresa alguien al equipo se crea una perdida de tiempo al capacitarla. Entre más gente entre menos responsabilidad.
4. La acumulación de tareas, hace que se pierda el feedback del cliente, porque no nos permite adaptarnos a la situación.
```

# Noveno principio

```
1. La atención continua a la excelencia técnica y al buen diseño mejora la agilidad. 
2. Desde el primer momento se debe buscar la excelencia en todo momento, no creer que se puede resolver las cosas a cada paso. 
3. Que el producto sea estable y AGRADABLE con el entorno.
```

# Decimo principio

```
1. La simplidad, o el arte de maximizar la cantidad de trabajo no realizado, es esencial.
2. No es necesario complicar de manera excesiva de presentar información.
3. Tanta planeación y crear documento predictivos solo son acumulaciones de poca utilidad.
4. El arte de hacer el trabajo más importante de una manera simple, saber claramente en que se esta trabajando. Los puntos claves.
5. Mantener las cosas lo más simple posible. En todos los sentidos.
```

# Undecimo principio

```
1. Las mejores arquitecturas, requisitos y diseños emergen de equipos autorganizados.
2. Busque el transfonodo de las necesidades del cliente y cree las soluciones.
3. Pedir al equipo que resuelva una solución y ver cómo ellos deciden cómo resolverlo. Así los miembros del equipo adoptando compromismo mayores.
4. Al trabjar de manera colaborativa entre varias personas de diferentes áreas fluye la retroalmientación.
5. Dar soluciones que el cliente necesite, no lo que pide.
```

# Duodecimo principio

```
1. A intervalos regulares el equipo reflexiona sobre cómo ser más efectivo para a continuación ajustar y perfeccionar su comportamiento en consecuencia.
2. Cada cierto tiempo recibir feedback del stakeholder y juntarse con el equipo para mejorar los procesos.
3. Existen técnicas de reflexión para mejorar ese rumbo y ser más efectivos como equipo.
```
