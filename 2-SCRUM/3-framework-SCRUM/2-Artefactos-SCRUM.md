# El producto Backlog (PB/Pila de producto)

```
1. Es uno de los artefactos dentro de SCRUM, que es una lista de tareas que hay que realizar.
2. Cuando se realizaa debe ser:
-Detallado: Se explica con suficiente detalle lo que se debe de hacer.
-Estimado: El equipo SCRUM estima la duración de CADA tarea.
-Emergente/Adaptable: A medida que va pasando el proyecto, se va mejorando el PB hacia lo que aporta valor. Es dinamico y cambia de manera permanenete. 
-Priorizado: En los puestos superiores se coloca lo que aporta más valor.
3. EL PO esta acargo del PB, es la máxima autoriadad sobre el PB. Todos pueden agregar tareas, pero quien lo prioriza o acepta es el PO. Debe tener mucho conocimiento del negocio.
4. No hay persona que pueda discutir con el PO, última palabra la tiene este muchacho.
5. Product goal: Es la meta del producto y sirve para tener en mente lo que se quiere construir. Cumple con los siguientes criterios:
-Especifico
-Medible
-Alcanzable
-Relevante
-Con limites de tiempo

Esta no se cambia hasta cumplir el objetivo.

6. El multitasking debe estar eliminado. Genera retrasos innecesarios.
```

# Creando un Product Backlog (PB)

```
Hay distintas manera de configurar esta lista. Pero cada item de nuesto PB se determinará PBI.

-Solo con tareas Ëpicas: Son historias de usuario grandes, que requieren varios Sprints. Son las que requieren más esfuerzo.
-Historias de usuario: Tener en cuenta muchos factores, que sea útil, bonito y vendible. Frente orientación al cliente. Creación de bienes y servicios.
-Tareas técincas: Si se habla el mismo lenguaje (como el de programación o marketing) se puede implementar este tipo de tareas.
```

# Principio de pareto

```
1. Sirve para la priorización, donde el primer puesto es el más importante y resuelto de manera prioritario.
2. Para las historias de usuario se tienen estas caracteriasticas:
-US: A, Valor total: 9, Prioridad 1, color: verde
-US: B, Valor total: 7, Prioridad 2, color: verde 
-US: C, Valor total: 2, Prioridad 3, color: amarillo
-US: D, Valor total: 2, Prioridad 4, color: amarillo
-US: E, Valor total: 1, Prioridad 5, color: rojo


3. Principio de pareto dice que el 20% de las causas tiene el 80% de efecto.
4. Con el 20% del US ya se debió crear el 80% del valor del producto 
5. Trabajar bajo puntos, no bajo horas. Las horas varían, pero los puntos no.
```

# Creando historias de usuario

```
1. Cada PBI debería tener detallado: La estimación, la descripción, el orden y el valor que simbolizan.
2. Requiere que cada item sea considerado para el público al que es llevado. El feedback es muy necesario.
3. Una buena US esta configurada de la siguiente manera:
-Cómo: Quien va a realizar la acción
-Quiero: Que va a realizar esa persona
-Para: Lo que se quiere lograr

ej:
-Cómo Estudiante del curso
-Quiero aprender mucho en este curso
-Para poder certificarme y tener un mejor trabajo

4. No entrar en demasiado detalle. Son guias sin ser del todo restrictivas. Tener pensamiento desde el punto de vista del cliente o usuario al que llegará el producto o servicio.

5. Generar un evento disruptivo e innovador, desde el mismo US.
```

# Técnica de priorización MoSCoW

```
1. Priorización tipo MosCoW:
-Mo: Must have/Tenemos que
-S: Should have/Deberíamos
-Co: Could have/Podríamos
-W: Won't but would have/No ahora, pero... 

2. Mo: Se ataca lo primero lo más prioritario. Se hace de totazo.
3. S: Se ataca lo que se debería hacer.  Se debe hacer lo antes posible.
4. Co: Posibles situaciones que se deben analizar. Se puede hacer hoy si se termino con lo anterior.
5. W: No es prioritario, pero se puede hacer después porque no representa un problema para hoy. No se incluye en el Sprint, pero se tiene un concideración.

6. Es muy normal que varien todas las posiciones, menos la uno. Eso se tiene que hacer o F.
```

# Técnina de priorización Kano

```
1. Dividir los items con respecto a 5 caracteristicas:
-Atractivo: Es una caracteristica que cuando esta presente es un elemento de valor increible, pero si no estuviera no ocasiona nada en el producto ni la relación que el consumidor tiene. Es un elemento diferenciador indiferente, el producto sigues siendo funcional. Si se tiene el tiempo se haga, si no se tiene el tiempo, dejarlo para el futuro.
-Unidimensional: Es una caracteristica que de estar el cliente va a estar feliz, pero de no ofrecer lo que se espera, el cliente se enojara. No son especificas para que el producto funcione, pero puede afectar severamente si no esta.
-Requerida: Son las caracteristicas que se espera que esten obligatoriamente y que si no están si es un problema mayor. Se espera que tenga algo, el cliente da por sentado que eso estará. Empezar por este tipo de caracteristicas.
-Indiferente: Al cliente les dará igual si están o no están ahí. Funciones que pueden ser descartadas.
-Inversa: Son caracteristicas que están ligadas 100% al usuario, a alguno les va a gustar que estén estas caracteristicas y a otros no. Se debe saber segmentar dependiendo del público que se quiere. 
```

# Estimación del PB

```
1. Estimación absoluta: tiempos muy bien defnidos, por el cliente o por el PO. Se carece de flexibilidad, se carece de feedback y de adaptabilidad. 
Ej: Diagrama de Gantt. Son etapas correlativas.
2. Estimación relativas: Son las indicadas para métodologias agiles y existen diversas técnicas. No se debe estimar por horas/días, ya que hay factores que retrasen o adelanten los tiempos planteados. Al momento de estimar es más facíl estimar cuando se sabe la proporción del proyecto. Se estima mejor por puntos de esfuerzo y no se incumplen fechas (aunque siempre ocurrira).
2.1 Nos permite ajustar con respecto a lo que va sucedio, por ej: una casa puede costar 10 puntos con 10 personas trabjando en ella, pero si la mitad del equipo se enferma, ahora costara 20 puntos poder completar la tarea.
2.2 Solo estima el que va a desarrollar la tarea. Ellos dan la presición de cuanto tomara, no el PM.
ESTIMA EL QUE HACE LA TAREAAAAAAAAAAA 
```

# Técnica de estimación Talles

```
1. Sigue este orden de estimación:
XS < S < M < L < XL

2. Una forma de estimar es tomar la más pequeña, la más fácil, darle la talla XS y establecer la talla de cada una de las otras tareas.
3. Esta técnica es visual y simple. Parece un juego.
4. Se agarra cada una de las tareas y se pone dentro de las camisas con respecto a las habilidades de cada miembro del equipo que trabajara en esa tarea. Luego se busca el punto medio.
```

# Técnica de estimación Planning Poker

```
1. Es basado en la secuencia de Fibonnacci
2. A cada uno de los miembros se le da a los participantes cartas, con las enumeraciónes de la secuencia Fibonnacci 1,2,3,5,8,13. El poyect manager menciona la tarea y todos los miembro voltean a la vez la dificultad que ellos creen realizar esta tarea.
3. Se encuentran 2 tipos de situaciones, Se encuentran resultados similares (1-2-3  QUE SEA SECUENCIAL), se suma y se le saca el promedio, aquí es 2. Si no son secuenciales (1-5-13 NO ES SECUENCIAL y hay mucha diferencia), el que puso el número más bajo comenta porque es facíl y el más alto comenta porque es dificil. Se vuelve a realizar la votación, se repite con respecto al resultado. Si sigue habíendo resultados no secuenciales, se descarta el más bajo y el más alto y se deja los del medio. Reodndear el resultado.
```

# El sprint backlog

```
1. Los ciclos de trabajos se llama sprint, y el sprint backlog son las tareas que se deben desarrollar durante este sprint
2. Se siguen los criterios SMART:
-Especifico: Algo lo más concreto posible, que las personas sepan que se esta necesitando. La idea es motivar a las personas.
-Medible: Que se tenga determinadas caracteristicas que se puedan comprobar. Producto goal medible para poder felicitar al trabajo por el equipo que hicieron.
-Alcanzable: Poner metas que sean posibles, porque si no es contraproducente.
-Relevante: Entregar productos de valor MVP
-Limites de tiempo: Orientados por el sprint
```

# Sprint Goal

```
1. Es la meta del sprint. Tiene que estar directamente relacionado al product goal y tiene que aportar a que se realice este product goal.
2. Requiere retroalmientación.
3. Busca generar impacto en cliente y negocio. Debe aportar valor.
4. Es una medida relativa del éxito del sprint.
```

# Creando el Sprint backlog

```
1. Forma parte de las tareas que se van a realizar durante el sprint.
2. Se deben realizar los PBI (Puede estar hecho por medio de US, Tareas, etc) que son todas las del sprint.
3. Se toman los primeros PBI que son necesarias y seleccionado por los desarrolladores (Las estimaciones las hacen quienes realmente van a realizar el producto), y se dividen en tareas.
4. Permitir a las personas que crean (Los devs), que van a crear, que y cuantos PBI van a utilizar y las tareas que requiere cada PBI.
```

# Incremento

```
1. Se tiene un product goal (Que no se ha desarrollado), entonces se crea el sprint 1 donde se pasa de 0 a 1, en el Sprint 2, es de 1 a 2. Y así sucesivamente, todo se va en incremento. Lo creado en el sprint actual + el resultado de los productos anteriores.
2. El incremento tiene un compromiso que se debe cumplir que es el DoD, la definición de terminado. 
3. Solo se considera incremento los puntos, tareas o productos terminados establecidos en el DoD (Definition of done).
4. Un buen incremento tiene estas caracteristicas:
-Se acerca al producto goal
-Debe aportar valor
-Deber ser funcioanl
-DoD es su compromiso
5. El incremento se presenta y se evalúa al final del Sprint. Es un evento que se le muestra al cliente que valide lo que esta pasando. 
```

# Definition of ready (DoR)

```
1. El PB -> tiene PBI
2. Antes de incluir antes de meterlos al SB entiendan esos PBI, sepan que es lo que se quiere, de que manera, cómo se debería hacer.
3. Se pasa por un filtro llamado (DoR). El equipo debe saber cómo desarrollar la tarea, si no sabe cómo hacerlo se retrasaran y pondra en peligro lograr el sprint goal.
4. El DoR es un acuerdo de todo el equipo SCRUM. La idea es que se saquen dudas y se sientan listos de comenzar con la tarea.
5. El equivo define el DoR, cuando ya se sienten listos pa darle.
6. Lo definen las personas que tienen conocimiento en la materia, ellos dan la pauta y las guias.
7. Cuando los desarrolladores entienden que deben hacer en cada PBI.
```

# Criterios de aceptación

```
1. Estan muy ligados al US
2. Los criterios de aceptación son una lista de requisitos que determinado función o tarea debería cumplir para ser considerada terminada como parte del producto.
3. Estos requisitos dan los cirterios.
4. El US pasa por ciertos requisitos que permitan entender el alcance, la funcionalidad y el porque de esta historia de suaurio, se llevan a los criterios de aceptación que deben ser entendidos por los miembros del equipo para considerar que seran cumplidos de manera satisfactoria. Esto hara que los PBI puedan desarrollarse de manera efectiva al realizar el Sprint Backlog.
5. Asi podemos tambien entender las limitaciones de los requerimientos que tiene cierto US.  
```

# Definition of done (DoD)

```
1. Terminado es al 100%, que ya quedo HECHO. Con posibilidad de uso del 100%
2. Como determinar cuando podemos decir que un PBI ya esta completado.
3. Se acuerda con el equipo para determinar que ya todo quedo listo. Cuando un PBI ya cumplió con todos los criterios de aceptación propuesto y negociasdos con el PO.
4. Una vez listos los criterios se puede considerar hecho, es decir, esta listo para ser utilizada.
5. La tarea ya cumple con los requisitos? Si si, se le añade al prototipo que será presentado al cliente y se chequea de la lista. Para poder obtener un feedback de su parte.
6. La definition of done deber ser al principio poco estricta y después cada vez más estricta.
7. No todos los PBI comparten los mismos definition of done.
8. No despedasar el producto, si no que se tengan ciertos criterios para poder obtener la difinición de producto terminado coherente con la uniformidad del producto.
```

# Sprint Burndown Chart

```
1. Evaluar el progreso. Que sea transparente y a simple vista.
2. Es un mapa de progreso es basado en dos ejes, en el Y se pone los puntos totales del Sprint que se determina de acuerdo a la velocidad del equipo. Ellos deciden cuanto son capaces de cumplir. Se establece al punto toal del sprint. Luego en el X se pone la duración del Sprint (No debe ser más de un mes) en días de trabajo (Puede ser diario o cada dos días).
3. Se actualiza este burndown Chart en la reunión diaria.
4. Se pone una linea recta que vaya desde el punto máximo de tareas en el día cero hasta ninguna tareá en el último Sprint, Así: \
5. Si se esta por encima de la linea recta, se esta excedidos en el tiempo de trabajo cuando se esta por debajo de la linea se esta yendo más rápido.
6. En caso de que se este muy por debajo de la linea, se pueden consumir otros PBI. 
7. Estos graficos nos ayuda a adaptarnos y la idea no es sobrecargar al equipo. Pero la idea no es ir alternado la duración del sprint.
8. Vaya tanteando y afine sus metodologias y equipos.

Ej: https://www.researchgate.net/profile/Marek-Milosz/publication/235675015/figure/fig1/AS:299921384198152@1448518238586/Detailed-Sprint-Burndown-Chart.png
```
